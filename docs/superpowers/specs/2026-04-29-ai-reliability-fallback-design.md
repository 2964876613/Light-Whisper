# AI 多模态稳定性与降级回退设计

## 1. 目标与范围

本设计用于解决光语 App 在加强结构化多模态链路后，用户频繁感知到“网络不佳”的问题。

核心目标：
- 将“真实网络失败”与“模型返回不合规”明确区分
- 保留现有结构化解析能力作为优先通道
- 当 JSON 不规范时，尽量回退到可播报短句，而不是直接失败
- 单次解析页与实时模式都尽量保持“有声反馈”
- 不引入重试，不显著增加时延

非目标：
- 不重做整套提示词体系
- 不新增多轮记忆或复杂缓存机制
- 不修改首页交互、ASR 主链路、TTS 主链路
- 不追求服务端 100% 严格结构化输出

## 2. 问题定义

当前实现中，`lib/services/doubao_api_service.dart` 的结构解析较严格，只要模型返回的 JSON 存在字段缺失、局部枚举不合法、低置信度或夹带解释文本，结构解析就可能返回空结果。

而页面层目前会把这类“请求成功但结构失败”的场景，和真正的超时、断网、鉴权失败一起折叠成“网络不佳”，导致：
- 用户误以为都是网络问题
- 实际已经有可读内容时仍然没有播报
- 实时模式的连续反馈被中断

问题本质不是单纯网络变差，而是当前链路把“结构化失败”过度放大成了“整体失败”。

## 3. 设计原则

1. 先保住可用，再保住结构完整。
2. 结构化结果是优先通道，但不是唯一真相源。
3. 只有真实请求异常才使用网络类兜底文案。
4. 只要服务端返回了可清洗、可裁剪、可播报的文本，就尽量给用户播出来。
5. 单次解析页可以展示更多结构信息；实时模式优先低延迟和连续反馈。

## 4. 总体方案

### 4.1 三层结果链

服务层图片解析改为三层处理链：

1. 请求层
- 判断是否发生真实请求失败
- 包括：超时、断网、401/403、5xx、请求异常
- 只有这一层失败才归类为网络问题

2. 结构层
- 请求成功后，优先尝试解析结构化 JSON
- 若得到合规结构结果，则继续走现有结构化展示和播报链路

3. 文本降级层
- 当请求成功但结构不合规时，不直接判空
- 改为从模型原始返回中提取一段可播报短句
- 若能提取到可用短句，则返回文本降级成功
- 只有清洗后仍为空，才判定为识别不稳定

### 4.2 统一结果模型

新增统一结果模型，替代当前“结构结果 + networkError 布尔值”的二元表达。

建议字段：
- `kind`：`structuredSuccess | textFallbackSuccess | networkFailure | emptyResponseFailure`
- `ttsText`：最终播报短句
- `structured`：仅在结构成功时存在，类型为 `StructuredVisionResult`
- `rawText`：模型原始文本，供降级和日志使用

## 5. 服务层改动设计

### 5.1 `lib/services/doubao_api_service.dart`

在现有请求流程基础上，新增统一结果模型与统一解析入口。

建议处理顺序：
1. 发起请求，拿到响应
2. 提取原始文本 `parsed`
3. 若 `parsed` 为空，返回 `emptyResponseFailure`
4. 尝试结构化解析 `_parseStructuredResult(parsed)`
5. 若成功，返回 `structuredSuccess`
6. 若失败，进入文本清洗与裁剪
7. 若得到非空短句，返回 `textFallbackSuccess`
8. 若清洗后仍为空，返回 `emptyResponseFailure`
9. 若请求抛出异常，则返回 `networkFailure`

### 5.2 文本清洗规则

文本降级不做复杂 NLP，只做本地轻量清洗：
- 去掉 markdown 代码块包裹，如 ```json
- 去掉首尾空白和多余空行
- 合并多行空白为单行空格
- 去掉明显的前后缀说明性文本
- 裁剪到适合 TTS 的长度，建议 20 到 30 字
- 若清洗后为空字符串，则视为不可播报

### 5.3 兼容现有公开方法

保留现有方法名，减少页面改动范围：
- `analyzeImage(...)`
- `analyzeImageStructured(...)`

但它们底层统一改为基于新的结果模型工作：
- `analyzeImage(...)` 返回最终可播报短句
- `analyzeImageStructured(...)` 不再单纯依赖结构解析成败，而是从统一结果模型中提取结构成功结果

必要时可新增一个更底层的方法，例如：
- `analyzeImageWithFallback(...)`

由页面层决定是消费完整结果，还是只消费 `ttsText`。

## 6. 校验放宽策略

### 6.1 必须继续严格的部分

以下规则仍然保留严格校验：
- 顶层必须是可解析对象，不能完全无法解码
- 最终必须能产出一段非空可播报文本
- 真实请求异常必须单独归类为 `networkFailure`
- 结构化成功时，核心枚举字段仍需校验，例如 `safety.level`

### 6.2 可以放宽的部分

以下规则从“整包判死”改为“局部跳过或降级”：
- `hazards` 可缺失、为空，或包含部分不合法元素
- `ocr_focus` 可缺失、为空，或包含部分不合法元素
- `traffic_light` 缺失时，不影响整体结果，只是不展示相关提示
- `summary` 不再作为结构成功的硬门槛
- 低置信度不再等于整个请求失败，而是触发不展示结构提示
- `tts_text` 过长时本地裁剪，不因超长直接失败

### 6.3 新的判断口径

旧口径：任何局部不合规 -> 整次失败。

新口径：
- 能保住结构 -> 返回结构成功
- 保不住结构但能保住文本 -> 返回文本降级成功
- 文本也保不住 -> 返回识别不稳定
- 只有真实请求异常 -> 返回网络失败

## 7. 页面消费规则

### 7.1 `lib/screens/chat_screen.dart`

单次解析页按结果类型分别处理：

- `structuredSuccess`
  - 展示 `_aiResult`
  - 展示 `_safetyHint`
  - 展示 `_contextHint`
  - 保持当前最完整体验

- `textFallbackSuccess`
  - 展示并播报主文案
  - 清空 `_safetyHint`
  - 清空 `_contextHint`
  - 不拼接不完整结构，避免误导

- `networkFailure`
  - 显示网络类兜底文案

- `emptyResponseFailure`
  - 显示识别不稳定类兜底文案

### 7.2 `lib/screens/live_vision_screen.dart`

实时模式优先“持续有反馈”：

- `structuredSuccess`
  - 正常更新 `_latestResult` 并播报

- `textFallbackSuccess`
  - 同样更新 `_latestResult` 并播报

- `networkFailure`
  - 播放网络类兜底文案

- `emptyResponseFailure`
  - 播放识别不稳定类文案

实时模式不依赖结构字段展示额外 UI，只消费最终短句，保持低延迟和连续反馈。

## 8. 超时与时延策略

本次不引入重试。

原因：
- 用户已明确选择不重试
- 实时模式更看重节奏连续性，而不是单次成功率最大化
- 作业场景下，优先解决“假网络失败”比堆叠复杂容错更重要

可选优化：
- 适度放宽当前请求超时配置，使慢响应少量回流
- 但不应因为等待更久而破坏实时模式节奏

## 9. 测试与验收

### 9.1 真实网络失败

验证点：
- 断网、超时、鉴权错误、服务端错误时
- 单次解析页与实时模式都只出现网络类文案
- 不误播为正常识别结果

### 9.2 请求成功但 JSON 不规范

验证点：
- 模型返回文字，但夹带解释文本、字段缺失、格式不严
- 单次解析页仍展示主文案
- 实时模式仍继续播报
- 结构提示区不展示

### 9.3 请求成功但内容为空

验证点：
- 返回空串、仅符号、清洗后无有效文本
- 进入识别不稳定类文案
- 不误报为网络问题

### 9.4 结构完全合规

验证点：
- 保持当前最佳体验
- 风险等级、红绿灯、障碍提示正常显示
- 说明本次改动未破坏原有结构化能力

## 10. 实施顺序

1. 在服务层新增统一结果模型
2. 改造请求返回链路，区分网络失败、结构成功、文本降级、空响应失败
3. 新增原始文本清洗与裁剪逻辑
4. 调整 `ChatScreen` 的分支消费规则
5. 调整 `LiveVisionScreen` 的分支消费规则
6. 做四类场景回归验证

## 11. 风险与应对

风险 1：文本降级播报不够规整
- 应对：本地清洗 + 长度裁剪，优先保证简短可听

风险 2：放宽校验后错误结构混入 UI
- 应对：仅结构成功时展示结构提示；降级文本不展示风险/红绿灯提示

风险 3：服务层返回语义变化导致页面分支遗漏
- 应对：统一结果模型收口，页面只按 `kind` 分支消费

## 12. 结论

本方案不否定现有 structured JSON 能力，而是把它从“唯一成功标准”调整为“优先成功通道”。只要服务端返回了可读内容，就尽量给用户播报；只有真实请求异常才提示网络问题。这样可以在不引入重试和不扩大代码量的前提下，明显改善“经常网络不佳”的体感问题，更适合作业场景下的稳定性优化。
